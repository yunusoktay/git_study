## Outcome

Day 11 introduces asynchronous programming in Dart.

### Synchronous vs Asynchronous

- **Synchronous**: code runs line by line. Each line waits for the previous to finish.
- **Asynchronous**: a long-running task can be started, and the program continues with other work. When the task finishes, its result is delivered later.

### The Event Loop

Dart runs on a single thread and uses an **event loop** to manage async work:

1. The main code runs first (synchronous part).
2. Asynchronous work (timers, I/O, `Future.delayed`) is queued.
3. When the main code is done, the event loop picks up ready tasks and runs them.
4. Callbacks attached via `.then(...)` are executed when their `Future` completes.

This is why the "After Future" line prints **before** "Future completed: ..." in the demo — the delayed task waits in the queue, the main code keeps running.

### Future

A `Future<T>` represents a **value that will be available later**. It can complete with:

- a value of type `T`, or
- an error

You attach callbacks to a `Future` using `.then(...)` to handle the value once it's ready.

```dart
Future<String> fetchData() {
  return Future.delayed(
    const Duration(seconds: 1),
    () => 'Data received!',
  );
}

fetchData().then((result) => print(result));
```

## Key Concepts

| Concept | What It Means |
|---|---|
| **Synchronous** | Runs in order, blocks until done |
| **Asynchronous** | Doesn't block, result comes later |
| **Event loop** | Mechanism that schedules async work |
| **`Future<T>`** | A placeholder for a value that arrives later |
| **`.then(...)`** | Runs when the Future has a value |
| **`Future.delayed`** | Produces a Future that completes after a duration |

## Why It Matters

In a mobile app (Flutter), the UI runs on a single thread. A slow synchronous task (network request, file read) would freeze the screen. Async programming lets those tasks run in the background so the UI stays responsive.
